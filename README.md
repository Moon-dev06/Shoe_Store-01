ส่วนที่ 1: โครงสร้างโปรเจกต์ (Frontend, Backend & K8s)
บรรยายขณะโชว์ภาพ **ไฟล์งาน.png** หรือหน้า VS Code

- ภาพรวมโครงสร้าง : ในโปรเจกต์ **Shoe_Store-01** เราแบ่งโครงสร้างงานออกเป็นสัดส่วนชัดเจนภายใต้แนวคิด Microservices

- หน้าบ้าน (Frontend) : ในโฟลเดอร์ `refined-app/frontend` จะประกอบด้วยไฟล์ `index.html`, `script.js` และ `style.css` ซึ่งทำหน้าที่เป็นส่วนแสดงผลร้านค้าที่คุณเห็นในหน้าเว็บ
- หลังบ้าน (Backend) : ส่วน `refined-app/backend` จะมีไฟล์ `server.js` เป็นหัวใจหลักในการจัดการ API และเชื่อมต่อกับ **MongoDB Atlas** โดยใช้ค่าคอนฟิกจากไฟล์ `.env`
- โครงสร้าง Deployment (K8s) : เรามีโฟลเดอร์ `k8s` ที่เก็บไฟล์ YAML สำคัญ เช่น `frontend-deploy.yaml` และ `backend-deploy.yaml` รวมถึง `hpa.yaml` ที่ใช้สำหรับทำ Auto-scaling บน Kubernetes

ส่วนที่ 2: การเพิ่ม Feature ใหม่ด้วยระบบ CI/CD
บรรยายประกอบวิดีโอ **เพิ่ม Feature.mp4** และ **deploy-newfeature.mp4**

- การอัปเดตโค้ด : จากวิดีโอเราได้ทำการเพิ่มฟังก์ชัน **Checkout Alert** เพื่อสร้าง Pop-up ยืนยันการชำระเงิน ซึ่งเป็นการยกระดับ User Experience ให้ดีขึ้น
- Automated Workflow : ทันทีที่เราทำการ `git push` โค้ดที่แก้ไขแล้ว ระบบ **GitHub Actions** (ที่ตั้งค่าไว้ใน `deploy.yml`) จะเริ่มทำงานโดยอัตโนมัติ
- การ Deploy แบบไร้รอยต่อ : วิดีโอ `deploy-newfeature.mp4` แสดงให้เห็นว่าระบบทำการ Build และ Update ลงใน Kubernetes Cluster ทันที โดยที่เราไม่ต้องเข้าหน้า Server ไปจัดการเอง นี่คือหัวใจของ CI/CD

ส่วนที่ 3: การทดสอบ Scaling และเสถียรภาพระบบ
บรรยายประกอบวิดีโอ **k6.mp4** และการโชว์หน้า Terminal

- Load Testing : เราใช้เครื่องมือ **k6** ผ่านไฟล์ `load-test.js` เพื่อจำลองการเข้าใช้งานพร้อมกัน (Virtual Users) เพื่อทดสอบขีดจำกัดของระบบ
- การสังเกตการณ์ (Observation) : ในวิดีโอจะเห็นว่าเรายิงโหลดไปที่พอร์ต Backend (เช่นพอร์ต **31427**) และใช้คำสั่ง `kubectl get pods -w` เพื่อเฝ้าดูสถานะของระบบ
- การทำ Auto-scaling : จุดที่น่าสนใจคือเมื่อโหลดสูงขึ้น Kubernetes จะสั่งเพิ่มจำนวน Pods ของ Backend อัตโนมัติ (Scale Out) เพื่อรับมือกับจำนวน Request ที่เพิ่มขึ้น
- สรุปผลการทดสอบ : ผลลัพธ์ในวิดีโอโชว์ให้เห็นว่าอัตราการล้มเหลว (Failure Rate) เป็น **0.00%** แม้ระบบจะทำงานหนัก แสดงว่าโครงสร้าง K8s ที่เราวางไว้มีความเสถียรสูงมาก

[ส่วนที่ 1: จุดเริ่มต้นของระบบอัตโนมัติ]
"หัวใจสำคัญของการทำงานเริ่มต้นที่ฝั่ง Developer ครับ เมื่อมีการแก้ไขโค้ดเสร็จและทำการ Git Push ขึ้นไปที่ Repository จุดนี้เองคือจุดเริ่มต้นของกระบวนการอัตโนมัติทั้งหมด โดยระบบ CI/CD จะเข้ามาจัดการต่อทันทีเพื่อตรวจสอบโค้ดและเตรียมความพร้อมก่อนการ Deploy สู่ระบบจริงครับ"

[ส่วนที่ 2: การจัดการ Container และ Infrastructure]
"ในขั้นตอนนี้ เราใช้ Docker ในการแพ็กทั้ง Web Application และ API ให้กลายเป็น Docker Image เพื่อให้มั่นใจว่าแอปพลิเคชันจะรันได้เหมือนกันในทุกสภาพแวดล้อม จากนั้นเราใช้ Kubernetes หรือ K8s มาเป็นตัวควบคุมหลักในการจัดการ Container ทั้งหมด ไม่ว่าจะเป็นการเปิด-ปิด หรือการกระจายโหลดงานในระบบครับ"

[ส่วนที่ 3: เส้นทางของผู้ใช้งานและความเร็ว]
"สำหรับลูกค้าที่เข้าใช้งาน ประสบการณ์เริ่มจาก Amazon Route 53 ที่ทำหน้าที่แปลงชื่อเว็บไซต์ ตามด้วยการทำงานของ Amazon CloudFront ที่จะช่วย Cache ข้อมูลและกระจายเนื้อหา ทำให้ลูกค้าทั่วโลกเข้าถึงเว็บไซต์ของเราได้อย่างรวดเร็วครับ"

[ส่วนที่ 4: ความปลอดภัยและการเข้าถึงข้อมูล]
"เรื่องความปลอดภัยคือสิ่งที่เราให้ความสำคัญที่สุดครับ เราใช้ Amazon Cognito ในการจัดการระบบ Login และยืนยันตัวตน เสริมเกราะป้องกันการโจมตีจากภายนอกด้วย AWS WAF และควบคุมสิทธิ์การเข้าถึงอย่างละเอียดผ่าน AWS IAM ครับ"

"นอกจากนี้ Web Application ของเราจะไม่ติดต่อกับ Database โดยตรง แต่จะสื่อสารผ่าน API และ Middleware ที่บริหารจัดการโดย Amazon API Gateway เพื่อเพิ่มความปลอดภัยและควบคุม Request อย่างมีประสิทธิภาพ โดยมี Application Load Balancer คอยกระจาย Traffic ไปยัง Container ที่รันอยู่บน Amazon EC2 ครับ"

[ส่วนที่ 5: การจัดการข้อมูลและการติดตามผล]
"ในส่วนของข้อมูล เราเลือกใช้ Amazon RDS สำหรับข้อมูลหลักอย่างผู้ใช้และคำสั่งซื้อ ใช้ Amazon DynamoDB สำหรับข้อมูลที่ต้องการความเร็วสูง และมี Amazon ElastiCache มาช่วยลดภาระของ Database หลัก ส่วนรูปภาพสินค้าทั้งหมดจะถูกเก็บไว้อย่างปลอดภัยบน Amazon S3 ครับ"

"เรายังมีการเฝ้าระวังระบบตลอด 24 ชั่วโมงผ่าน Amazon CloudWatch และ CloudWatch Alarm ที่จะแจ้งเตือนทันทีเมื่อพบความผิดปกติ เช่น CPU พุ่งสูง หรือมี Container ตัวไหนล่มไปครับ"

[ส่วนที่ 6: ระบบการขยายตัวอัตโนมัติ (Auto Scaling)]
"สุดท้ายคือความสามารถในการยืดหยุ่นครับ ระบบของเราใช้ HPA หรือ Horizontal Pod Autoscaler เมื่อมีลูกค้าเข้ามาใช้งานจำนวนมาก Kubernetes จะสั่งเพิ่มจำนวน Container ให้โดยอัตโนมัติเพื่อให้ระบบไหลลื่น และเมื่อผู้ใช้งานน้อยลง ระบบก็จะลดจำนวน Container ลงเองเพื่อประหยัดค่าใช้จ่าย ซึ่งทั้งหมดนี้ทำงานอย่างชาญฉลาดตามปริมาณการใช้งานจริงครับ"

"นี่คือโครงสร้างทั้งหมดของ Refined ที่ทั้งรวดเร็ว ปลอดภัย และพร้อมรองรับการเติบโตครับ ขอบคุณครับ"
	🔁 ผมขอขยายความการทำงานของ Auto Scaling ให้เห็นภาพชัดๆ ดังนี้ : ในช่วงเช้า ที่มีลูกค้าน้อย ระบบอาจจะรัน Container ไว้เพียง 2 ตัว เพื่อประหยัดทรัพยากร
พอถึงช่วงเที่ยง ที่เรามีแคมเปญ Flash Sale และมีลูกค้าแห่เข้าพร้อมกันถึง 10,000 คน ระบบ Kubernetes จะสั่งการผ่าน HPA (Horizontal Pod Autoscaler) ให้เพิ่ม Container เป็น 20 ตัว ทันทีแบบอัตโนมัติ เพื่อรองรับ Traffic มหาศาลไม่ให้เว็บล่ม
และเมื่อเข้าสู่ช่วงดึก ที่ลูกค้าลดน้อยลง Kubernetes ก็จะสั่ง ลดจำนวน Container กลับมาเหลือ 2 ตัว ตามเดิม

บทสรุปการนำเสนอ
 จากภาพรวมการอัปเกรดเวอร์ชัน Final ของ Refined นะครับ เราได้ยกระดับ Infrastructure มาใช้ EC2 สเปก t3.medium ที่ทำ High Availability และ Auto Scaling เพื่อให้ระบบเสถียรตลอด 24 ชั่วโมง ด้านฐานข้อมูลเราเปลี่ยนมาใช้ Amazon RDS ที่ประสิทธิภาพสูงกว่า ทำงานร่วมกับ DynamoDB และ S3 เพื่อการจัดการข้อมูลที่รวดเร็ว ส่วนเรื่องความปลอดภัยเราใช้ Amazon Cognito ในการยืนยันตัวตนและ AWS IAM ควบคุมสิทธิ์อย่างละเอียด และที่สำคัญคือเราใช้ Kubernetes HPA ร่วมกับ CI/CD เพื่อขยายจำนวน Container อัตโนมัติตามการใช้งานจริง ทำให้ระบบลื่นไหลและประหยัดค่าใช้จ่ายที่สุดครับ
